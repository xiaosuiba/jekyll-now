Edge Edgex

在 Kubernetes 上运行 Edgex

Posted by Chris Li on Tuesday, July 9, 2019

EdgeX 官方目前使用了 docker-compose 进行安装，由于我们底层使用了 kubernetes 对所有的容器进行管理，所以希望将 EdgeX 跑在 Kubernetes。 官方目前的考虑是进一步做微服务的拆解，加强横向扩展能力，做到这一步之后再上 kubernetes。 目前可以在其官方的 blog 中找到一篇 huawei-runs-edgex-foundry-kubernetes/ 文章，作者已经成功进行了尝试。但是由于文章比较老，使用的还是 java 版本，而当前我们使用的是 go 语言的版本，所以借鉴其思路，我们在 edgex-go 0.7.1 上面进行了尝试。

EdgeX 在 docker-compose 和 kuberntes 上运行的最大的不同之处在于其服务的注册发现机制，官方本来采用了 consul 作为服务的发现注册中心，这在 docker-compose 中是必要的，但是由于 kubernetes 使用了 dns 作为自己的注册发现机制，所以我们要做的修改就是这一步。

docker-compose 文件可以使用 kompose 转换成为 kubernetes 的 manifests，我们也借助了这个工具，下面是我们的操作步骤。

1. 使用 kompose 工具转换 docker-compose.yml

首先，下载 kompose 工具：

$ curl -L https://github.com/kubernetes/kompose/releases/download/v1.18.0/kompose-linux-amd64 -o /usr/local/bin/kompose
$ chmod +x /usr/local/bin/kompose

然后，使用 kompose 对 docker-compose.yml 进行转换：

kompose 会将 docker-compose 的 service 转换为 deployment、service、volume 三类文件，我们打开一个 deploy 观察下：

apiVersion: extensions/v1beta1
kind: Deployment
metadata:
  annotations:
    kompose.cmd: kompose convert -f docker-compose.yml
    kompose.version: 1.18.0 (06a2e56)
  creationTimestamp: null
  labels:
    io.kompose.service: command
  name: command
spec:
  replicas: 1
  strategy:
    type: Recreate
  template:
    metadata:
      creationTimestamp: null
      labels:
        io.kompose.service: command
    spec:
      containers:
      - image: edgexfoundry/docker-core-command-go:0.7.1
        name: edgex-core-command
        ports:
        - containerPort: 48082
        resources: {}
        volumeMounts:
        - mountPath: /data/db
          name: db-data
        - mountPath: /edgex/logs
          name: log-data
        - mountPath: /consul/config
          name: consul-config
        - mountPath: /consul/data
          name: consul-data

      restartPolicy: Always
      volumes:
      - name: db-data
        persistentVolumeClaim:
          claimName: db-data
      - name: log-data
        persistentVolumeClaim:
          claimName: log-data

        persistentVolumeClaim:
          claimName: consul-config
      - name: consul-data
        persistentVolumeClaim:
          claimName: consul-data

由于 docker-compose 与 kubernetes volume 共用机制的不同，转换之后的 pvc 要进行符合自己需求的修改，这里我们使用 hostpath 的简单修改方式：

        volumeMounts:
        - mountPath: /data/db
          name: data-db
        - mountPath: /edgex/logs
          name: edgex-logs
        - mountPath: /consul/config
          name: consul-config
        - mountPath: /consul/data
          name: consul-data
      restartPolicy: Always
      hostname: edgex-core-command
      volumes:
      - name: data-db
        hostPath:
          path: /data/db
      - name: edgex-logs
        hostPath:
          path: /edgex/logs
      - name: consul-config
        hostPath:
          path: /consul/config
      - name: consul-data
        hostPath:
          path: /consul/data

修改需要用到的所有的 deployment。 转换之后的 service 默认使用 clusterIp 的类型，可以不用修改，如果需要外部访问，可以按实际情况修改为 LoadBalancer 或者 NodePort。

2. 修改组件，停止使用 consul 作为服务注册发现机制

EdgeX 的组件默认使用了 consul 作为服务注册发现机制，而 kubernetes 自带了 dns 的服务注册发现机制，所以我们需要关闭组件使用 consul。 截至 1.0.0 版本中，除 ruleengine 以外，其余组件都使用了 go 进行重写，其组件的命令行参数 --registry 用于控制启用 consul，打开任意组件的 dockerfile（例如 core-command）：

ENTRYPOINT ["/core-command","--registry","--profile=docker","--confdir=/res"]

可以看到默认启用了 --registry 参数，所以我们可以重新 build 镜像，或者直接在 deployment 的 manifest 文件中覆盖启动参数：

        command: ["/core-command","--profile=docker","--confdir=/res"]

完整的 deployment 文件如下：

apiVersion: extensions/v1beta1
kind: Deployment
metadata:
  labels:
    io.kompose.service: edgex-core-command
  name: edgex-core-command
spec:
  replicas: 1
  template:
    metadata:
      labels:
        io.kompose.service: edgex-core-command
    spec:
      containers:
      - image: edgexfoundry/docker-core-command-go:0.7.1
        name: edgex-core-command
        ports:
        - containerPort: 48082
        command: ["/core-command","--profile=docker","--confdir=/res"]
        resources: {}
        volumeMounts:
        - mountPath: /data/db
          name: data-db
        - mountPath: /edgex/logs
          name: edgex-logs
        - mountPath: /consul/config
          name: consul-config
        - mountPath: /consul/data
          name: consul-data
      restartPolicy: Always
      #hostname: edgex-core-command
      volumes:
      - name: data-db
        hostPath:
          path: /data/db
      - name: edgex-logs
        hostPath:
          path: /edgex/logs
      - name: consul-config
        hostPath:
          path: /consul/config
      - name: consul-data
        hostPath:
          path: /consul/data
status: {}

对于 ruleengine，需要修改其 /edgex/edgex-support-rulesengine/bootstrap.properties 文件，将 spring.cloud.consul.enabled 设置为 false。同样的，可以选择重新 build 镜像文件，或者使用 volume 挂载一个自定义的文件，我们这里使用 configmap 来作为 volume 进行挂载，完整的 yaml 如下：

---
apiVersion: extensions/v1beta1
kind: Deployment
metadata:
  labels:
    io.kompose.service: edgex-support-rulesengine
  name: rulesengine
spec:
  replicas: 1
  template:
    metadata:
      labels:
        io.kompose.service: edgex-support-rulesengine
    spec:
      containers:
      - image: edgexfoundry/docker-support-rulesengine:0.7.0
        name: edgex-support-rulesengine
        ports:
        - containerPort: 48075
        resources: {}
        volumeMounts:
        - mountPath: /data/db
          name: data-db
        - mountPath: /edgex/logs
          name: edgex-logs
        - mountPath: /consul/config
          name: consul-config
        - mountPath: /consul/data
          name: consul-data
        - mountPath: /edgex/edgex-support-rulesengine/bootstrap.properties
          name: bootstrap-properties
          subPath: bootstrap.properties
      restartPolicy: Always
      #hostname: edgex-support-rulesengine
      volumes:
      - name: data-db
        hostPath:
          path: /data/db
      - name: edgex-logs
        hostPath:
          path: /edgex/logs
      - name: consul-config
        hostPath:
          path: /consul/config
      - name: consul-data
        hostPath:
          path: /consul/data
      - name: bootstrap-properties
        configMap:
          name: ruleengine-config
          items:
          - key: bootstrap.properties
            path: bootstrap.properties
status: {}

---
kind: ConfigMap
apiVersion: v1
metadata:
  name: ruleengine-config
data:
  # example of a complex property defined using --from-file
  bootstrap.properties: |
      spring.application.name=edgex-support-rulesengine
      spring.cloud.consul.host=edgex-core-consul
      spring.cloud.consul.port=8500
      spring.cloud.consul.config.profileSeparator=;
      spring.cloud.consul.enabled=false
      spring.profiles.active=docker

最后，使用 kubectl 创建对应的 deployment 及 service 即可。

我将使用的文件及一个工具放到了 github 上，有需要的同学可以参考 这里 我使用了 0.7.1 的版本，请根据需要自行修改。 hack 目录中有一个小工具 edgex-up.sh，可以使用 .\edgex-up.sh start 和 .\edgex-up.sh stop 来分别开始及停止 EdgeX。
